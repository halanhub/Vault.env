## Case study: why we built it

Teams routinely juggle API keys and `.env` files across laptops, CI, and staging. The usual pattern is either **plain env vars in a chat** (bad) or **a single shared password on a document** (also bad). We wanted a product that feels like a polished app-first vault but keeps the cryptography where we can reason about it: **on the client**, before anything touches our backend.

**Vault.env** is that case study shipped as a product: a Next.js app with a **vault master password**, **per-project workspaces**, **encrypted notes and files**, and a **CLI** that speaks to the same ciphertext in Firestore so developers can `pull` a `.env` without exposing secrets to our application logic.

## How information moves (and what never should)

### 1. Master password and keys

- You choose a **vault master password** in the browser.
- We derive an AES-256-GCM key with **PBKDF2** (600,000 iterations, SHA-256) using **Web Crypto**.
- The derived key stays in **memory** (Zustand store) for the session. We do **not** send the master password to our servers to “unlock” your vault server-side.

### 2. Secrets and files

- Each secret **value** is encrypted with **AES-256-GCM**; we store **ciphertext**, **IV**, and **salt** in Firestore (and encrypted file bytes in Firebase Storage with IV/salt metadata).
- **TLS** protects data in transit between the browser and Google’s endpoints.

### 3. What Firebase can still see

- **Authentication**: Firebase Auth knows your **account identity** (email, uid). That is inherent to “log in to sync.”
- **Metadata**: Project names, secret **key names** (labels like `DATABASE_URL`), file names, and sizes are stored as **plaintext fields** so the UI can list and search them. Only the **values** (and file contents) are ciphertext in our model.
- **Trust boundary**: You trust **Firebase/Google** for availability and transport; you trust **Vault.env** for client code that runs in your browser. We do not run a custom server that decrypts your secrets for you.

![Layered architecture](public/readme/architecture-layers.svg)

### 4. CLI

- The **[@vaultenv/cli](https://www.npmjs.com/package/@vaultenv/cli)** package uses the same **`@vaultenv/crypto`** algorithms so `pull` / `push` round-trip the same payloads.
- Account login uses Firebase Auth; you can use **`--no-store`** if you do not want credentials in `~/.vault-env/`.

## Product snapshot

| Area | Behavior |
|------|----------|
| **Encryption** | AES-256-GCM, PBKDF2-derived keys, per-value salt/IV |
| **Lock screen** | Clears derived keys; idle timeout returns you to lock |
| **Projects** | Dashboard with encrypted secrets, files, and markdown notes |
| **Blog / marketing** | Static content; sitemap and robots tuned for public pages |

## Development

```bash
git clone <repo-url>
cd vault.env
npm install
cp .env.example .env.local
# Fill Firebase keys; enable Auth, Firestore, Storage; deploy rules from firestore.rules / storage.rules
npm run dev
```

- **Build**: `npm run build`
- **Crypto package**: `packages/crypto` (shared with CLI)
- **CLI workspace**: `packages/cli`
